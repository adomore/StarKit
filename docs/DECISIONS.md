- **D-009** 2026-07-16 (T0-1) — **Serialization policy for hash-pinned artifacts.** JSON: `serde_json::to_string_pretty` (2-space indent, LF, `ryu` shortest-round-trip floats — no locale, no platform-dependent `%g`), plus exactly one trailing newline; pinned by `emit::tests::float_json_formatting_is_pinned`. Manifest: `sha256sum` format (`<hex>  <path>`), paths **relative to the output root** with forward slashes, sorted byte-wise — so no absolute path can leak into a committed file and the manifest is identical regardless of where it was generated. A `.gitattributes` marks `fixtures/expected/** -text`: without it, a Windows checkout would rewrite LF→CRLF in the committed JSON and break every hash.
- **D-010** 2026-07-16 (T0-1) — **Photometric conventions for truth catalogs**, needed because FIXTURES.md schema v1 says only "linear units" and has no colour field. (1) `gain_adu_per_e = 1.0` in every suite, so electrons and ADU are numerically equal. (2) `flux`/`peak` describe the **noiseless, mean-of-RGB-channels** star signal, excluding background; per-star tint is normalised to mean 1.0 across channels, so a consumer measuring `(R+G+B)/3` sees exactly the catalog numbers — this is the contract T0-2's oracle must measure against. (3) `peak` is the **pre-clip** value: a saturated star's true peak is recorded even though the image clips at 65535. (4) Halo and bleed components on the `saturated` suite are *additional* signal and deliberately **not** included in `flux`, which stays the PSF core flux that detection actually measures. Alternative for (4) — folding halo flux into `flux` — was rejected: it would make the catalog disagree with any sane aperture photometry.
- **D-011** 2026-07-16 (T0-1) — **T0-1 acceptance tests are split by cost.** The default `cargo test` runs byte-identity on tiny ad-hoc params (covering all four artifact types incl. the sky-mask PNG) plus schema/population validation of the *committed* truth catalogs for all five real suites. Regenerating the five real suites (~10⁸ Poisson draws; 5m39s in release, far longer in debug) is `#[ignore]`d and runs via `cargo test --release -- --ignored`, wired into `ci.sh` at T0-4. Approved by the product owner before implementation. Rationale: a default test run that takes minutes stops being run. Both AC tests exist at full scale and were executed on the final code — they are gated, not skipped.
- **D-012** 2026-07-16 (T0-1) — **Determinism guarantee is same-toolchain/same-platform**, which is what INV-2 ("same input + same params ⇒ bit-identical output") and PRD §6 ("no wall-clock or thread-order dependence") actually require, and what the T0-1 AC tests. Cross-*platform* byte-identity is **not** guaranteed and is not achievable at reasonable cost: `rand_distr`'s Poisson/Normal samplers route transcendentals through the platform libm, so a Linux CI regenerating fixtures could produce different bytes from this Windows-generated manifest even at the same seed. Committed manifest hashes therefore pin *this* platform's output. **Open for T0-4:** either restrict the CI manifest check to the platform that generated it, or commit per-platform manifests, or (expensive) route every transcendental through a bit-exact `libm` and hand-roll the samplers. Flagged rather than decided, since it shapes ci.sh's design.
- **D-013** 2026-07-16 (T0-1) — **The `saturated` suite's bleed column carries an explicit stylization factor** (`halo.bleed_length_gain`, default 6.0). Context: FIXTURES.md asks for a "halo/bleed approximation", and the first implementation derived the column length physically from the star's overflow charge. Measured on the generated fixture, that produced a saturated extent of **5 px vertical × 5 px horizontal** — i.e. a column shorter than the already-saturated core, so `bleed: true` rendered identically to `bleed: false` while the committed params claimed otherwise. The cause is not a coding bug: at this suite's brightnesses (peaks ≈ 4–7 × full well) the physically implied column genuinely is that short — real blooming needs far more overexposure plus well geometry this generator does not model. Decision: keep the overflow-charge derivation, scale it by an explicit, honestly-named factor, and prove the effect with a test (`render::tests::bleed_elongates_the_saturated_core_vertically`, which asserts `bleed:true` elongates vertically vs `bleed:false` and does not widen horizontally). Result on the committed fixture: 27 px vertical × 5 px horizontal. Alternative — set `bleed: false` and drop the code — rejected: FIXTURES.md asks for the bleed, and its purpose is to give Phase-1 mask code saturated-star structure to cope with. This is a parameter default, hence decided rather than escalated; the *model* is unchanged.

- **D-017** 2026-07-16 (T0-2) — **Truth `snr` is PER-CHANNEL peak SNR; the definition is pinned in FIXTURES.md rather than changed.** Found while calibrating the oracle: T0-1 renders `snr = peak / sqrt(peak + bkg + read²)` using single-channel noise, while D-010 declares the catalog describes the *mean-of-RGB* plane. Averaging three channels cuts noise by exactly √3 (confirmed: predicted mean-plane background RMS 11.02 vs 10.95 measured), so the stored number is not the SNR of the plane a consumer measures. Escalated to the product owner rather than reinterpreted, because it changes what every SNR-conditioned AC means. **Decision: keep per-channel and document it loudly.** The two framings select identical stars — per-channel `snr ≥ 5` ≡ mean-of-RGB `snr ≥ 8.66` — so this is a labelling choice, not a difficulty one; but conditioning FR-2's existing "≥ 5" bar on the mean-of-RGB value makes it *unachievable at any threshold* (measured on basic-5k: recall peaks at 97.19 %, and only at 25.5 % precision). Alternatives: (a) switch to mean-of-RGB and restate the AC as ≥ 8.66 — equivalent but requires regenerating all five suites, the manifest and every committed catalog, and reads like moving the goalposts; (b) switch and keep "≥ 5" — rejected, it invents a bar the reference instrument cannot clear. **Cost, stated plainly:** "recall ≥ 98 % at SNR ≥ 5" is an easier achievement than it sounds, because those stars sit at ≈ 8.7 σ in the measured plane. FIXTURES.md and every emitted report now carry that caveat so the number cannot be quoted without it.
- **D-018** 2026-07-16 (T0-2) — **The oracle measures FWHM by an elliptical Gaussian2D fit, with a characterised bias, not by image moments.** Image second moments were the first implementation and are broken for this fixture family: measured +36 % against truth (and +56 % over the full matched population). Cause is not a coding bug — a Moffat's second moment is dominated by its wings and depends on β, which varies per star in [2.5, 4.5], so moments measure the wings rather than the half-max width. An astropy `Gaussian2D` + `TRFLSQFitter` fit is used instead: **+2.3 % on elliptical-Gaussian stars** (residual pixel-integration bias — the fixture integrates the profile over each pixel, the model is evaluated at pixel centres) and **+14.2 % on circular-Moffat stars** (model mismatch). Both are flat across fit-box sizes, i.e. explainable systematics rather than tuning artifacts. The suites are ~60 % Moffat so the reported figure (~10 %) is dominated by the Moffat term. Deliberately *not* tuned away by fitting Moffat-and-Gaussian and selecting by residual: that would match the instrument to the generator's exact PSF family and weaken the independence the oracle exists for. Acceptable because FWHM is in no T0-2 acceptance criterion, and T1-3's oracle cross-check is **recall only** — starkit-core's own "median FWHM error ≤ 10 %" is graded against *truth*, not against the oracle. Isophotal masking scored better (+4.65 %) but only by accident of the isophote cancelling the wing inflation, and it drifts with brightness; a lucky number was rejected in favour of an explainable one.
- **D-019** 2026-07-16 (T0-2) — **On the nightscape the oracle excludes the silhouette plus a kernel radius from detection.** Untreated, `nightscape-fg` scored **59 % precision** on 4175 false positives, 99.9 % of them within 20 px of foreground (median 2.8 px). The cause is not the background model and not the foreground texture — both were tested and cleared. At one such false positive the background estimate was 300.1, the data 298.3, i.e. a residual of **−1.8** against a threshold of 55, and DAOStarFinder fired anyway. Only the convolved image explains it: DAOFIND correlates a **zero-sum** kernel, so it responds to how much brighter a pixel is than its neighbourhood, and ordinary sky beside a dark tree silhouette wins that comparison outright. Fix: pass a detection mask of the foreground dilated by the DAOFIND kernel radius (`ceil(1.5 × fwhm / 2.355) + 1`, = 3 px at fwhm 3.2), so no kernel footprint can straddle the sky/silhouette edge. Result: **precision 59.06 % → 99.93 %**. Recall falls 95.74 % → 91.63 %, and that cost is fully accounted for: 4.89 % of SNR ≥ 5 truth stars lie inside the exclusion zone, versus the 4.11 pp observed drop. The oracle now declines to measure 3.5 % of the sky rather than reporting 4175 stars that are not there. `measure.py` therefore takes an optional `--sky-mask` beyond the interface in oracle/CLAUDE.md; for a nightscape it is a correctness requirement, not a convenience. **Note for Phase 1:** starkit-core will meet this same edge effect at the sky-mask boundary (T1-3/T1-5). It is a property of matched-filter detection, not of photutils.
- **D-020** 2026-07-16 (T0-2) — **Precision counts a detection on *any* real star as a success, not only one above the SNR floor.** FIXTURES.md says "recall and precision are computed over truth stars with peak SNR ≥ 5", which is well-defined for recall but ambiguous for precision, since precision is a property of detections. Reading it strictly — only SNR ≥ 5 matches count as true positives — would score a detector *worse* for correctly finding a real SNR 4 star, which inverts the metric's purpose; on basic-5k it reads 82.07 % against 99.87 % for the standard definition. Decision: `precision` = detections matched to any truth star / all detections. The strict variant is emitted alongside as `precision_strict` in every report so the choice is visible and reversible rather than baked into a single number, and both definitions ship in the report's `metric_definitions` block. Decided rather than escalated because the alternative is self-evidently broken; flagged here so it can be overturned cheaply.

The commands now invoke `python -m pip` / `python -m pytest` rather than the `pip` shim (works identically under both layouts) and carry an explicit note about the Windows path. `pytest>=8` joins `requirements.txt` under the existing loose-pin rule, since T0-2's acceptance criteria are encoded as tests and a test runner that is not installed by the documented setup is not a real gate. - **D-022** 2026-07-16 (T0-3) — **`serde_json` needs `float_roundtrip` + `preserve_order` for the catalog contract to hold.** Found by the T0-3 round-trip test, which failed on the very first committed catalog. Two independent causes, both invisible without a byte-level check. (1) **serde_json's default float parser is not correctly rounded.** `from_str::<f64>("93.07565673947249")` returns bits `…7067`; Rust's own `str::parse` (correctly rounded since 1.55) returns `…7068`. The serializer is fine — ryu is exact — so values written by `starkit-fixtures` came back **1 ULP low on 3702 of 65050 committed floats (~5.7 %)**. Enabling the `float_roundtrip` feature makes the parser exact. This silently invalidated D-009's round-trip claim: the JSON *was* shortest-round-trip on write, but could not be read back. (2) **`serde_json::Value` maps are `BTreeMap` by default**, so the opaque `generator` block re-serialized with keys alphabetised (`background, foreground, halo…` instead of `schema, suite, seed…`). `preserve_order` (via `indexmap`) fixes it. With both: **0 of 65050 lines differ**. New transitive deps `indexmap` 2.14, `hashbrown` 0.17, `equivalent` 1.0 — all `Apache-2.0 OR MIT`, INV-6 clean. **Consequence beyond T0-3:** any Phase-1 code that reads a catalog and writes it back (T1-8's "re-run produces hash-identical outputs") would have corrupted floats without this. Cost: `preserve_order` changes `Value` map behaviour workspace-wide, which is the intent, and `float_roundtrip` trades a little parse speed for correctness — an easy trade for a project whose second invariant is determinism.
- **D-023** 2026-07-16 (T0-3) — **Schema v1 gains an optional `measurement` block; this is the only change from the draft.** The oracle already emitted it (T0-2) but the draft schema had no such field, so `starkit-core` parsed measured catalogs and *silently dropped* their provenance. Approved by the product owner before the freeze. It is the mirror image of `generator`: a truth catalog says how it was generated (params + seed), a measured catalog says how it was measured (method, thresholds, exact library versions), and both are opaque JSON to `starkit-core`, which must not grow an opinion about either. A catalog carries one or the other, never both. Additive and optional, so it breaks no existing reader. Alternative — strip the block from the oracle's catalogs and keep provenance only in the reports — was rejected: `compare.py` already consumes it, and a catalog that outlives its report would then be unable to say what produced it, which is exactly the traceability oracle/CLAUDE.md demands. Phase 1's `starkit-core::detect` gets the same field for free. Everything else in the draft is frozen unchanged; `snr` stays required-in-truth/optional-in-measured, `tier` stays required (the oracle assigns tertiles purely to be schema-valid — evaluation matches by position, never by tier).

- **D-029** 2026-07-16 (T1-1) — **D-005 resolved: there is no ICC transform backend. Not lcms2, not qcms, not built-in curve tables — none is needed.** The working space is **linear light in the source's own primaries**: `starkit-io` inverts the embedded tone curve, leaves the primaries alone, and carries the ICC profile through byte-for-byte to re-attach on encode. Two reasons, neither stylistic. (1) **Converting to a canonical space destroys data.** ProPhoto RGB is far wider than sRGB; normalising a ProPhoto frame into linear-sRGB clips every out-of-gamut colour irreversibly, and wide-gamut is precisely why the photographer works there. (2) **It would break FR-1.** The no-op round-trip must be *pixel-identical*, and convert-then-convert-back is only approximately invertible. This is also enough: star reduction and enhancement are per-channel photometry, not colorimetry — nothing downstream needs a shared colour space, and Photoshop reads the preserved profile and sees correct colour. It satisfies INV-4 (linear light, conversion only at the boundary) and starkit-io/CLAUDE.md's "de-linearize through the same transform used at decode" literally: the same curve, inverted, taken from `Meta::trc` rather than re-derived, so the symmetry is structural. Implementation is a ~200-line parser for `rTRC`/`gTRC`/`bTRC` (`curv` gamma/table and `para` types 0-4, all folded to the type-4 general form) — **no FFI, no new dependency**. Round-tripping is proven exhaustively over all 65536 16-bit codes per curve type, not sampled: `icc::tests::*_round_trips_every_16_bit_value`. Should a feature ever genuinely need cross-profile conversion, it must first justify the gamut loss; revisit this entry then.
- **D-030** 2026-07-16 (T1-1) — **EXIF preservation is complete for TIFF→TIFF and absent for JPEG/PNG→TIFF. Stated here rather than left to be discovered.** The two formats mean different things by "the EXIF". In a **JPEG** it is a self-contained APP1 blob carrying its own TIFF header with internally-relative offsets — captured verbatim. In a **TIFF** it is a sub-IFD woven into the host file, where any value over four bytes lives at an *absolute file offset*; copying those bytes into a new file would produce pointers into unrelated data. So `exif.rs` resolves a TIFF's Exif IFD into typed values and the encoder rewrites them with correct offsets (`extra_directory` + tag 34665). **Gap:** a JPEG's or PNG's captured blob is *not* written into the output TIFF — splicing it in would require rebasing every internal offset, which is not implemented. It is dropped, not written broken. Justification: the PRD's stated workflow (§2) is stacked, stretched **16-bit TIFF**; TIFF→TIFF is the path that carries EXIF worth keeping, and it works. JPEG/PNG are secondary inputs whose EXIF is camera metadata about a file that is already several processing steps removed. Also **not** copied even on the TIFF path: GPS IFD (34853), Interop IFD (40965) and MakerNote (37500) — all pointer-valued or offset-dependent, so a copy would produce a dangling offset, which is worse than an honest omission because a reader would follow it. Revisit if the photographer reports a JPEG/PNG EXIF need; the fix is an offset-rebasing IFD writer, bounded but not free.
- **D-031** 2026-07-16 (T1-2) — **The fixtures are linear but untagged, so `starkit-io` mis-linearizes them. Flagged now, decided at T1-3.** Found while wiring the first `starkit-core` algorithm to real data. The fixture TIFFs declare `color_space: linear-rgb` in their truth catalog and are linear by construction, but they carry **no ICC profile** — so `starkit-io` correctly follows its own rule (untagged ⇒ assume sRGB, D-029/starkit-io CLAUDE.md) and applies an sRGB inverse curve to data that never had one. The decoded values are then not the ADU the truth catalog is written in, and any photometry graded against truth would be wrong. **Not a bug in either component**: the fixture is honest, the decoder is honest, and the two have simply never met until now. It does not affect T1-2, whose test reads the fixture TIFF directly (INV-7 permits it) and is graded in ADU — the same units the oracle works in, since photutils' `tifffile` also reads raw samples. **It will bite at T1-3 and T1-8**, where detection and the CLI must go through `starkit-io` and be graded against truth. Options, in rough order of preference: **(a)** have `starkit-fixtures` embed a linear (gamma 1.0) ICC profile — most principled, since the file would then say what it is, but it regenerates all five suites, the manifest and every oracle report; **(b)** give `starkit-io` an explicit "source is linear" override — cheap, but adds a way to lie to the decoder; **(c)** keep grading core against raw samples and never route fixtures through `starkit-io` — rejected, it would leave the end-to-end path untested exactly where FR-1 meets FR-2. Deliberately **not** decided here: T1-2's scope is the background estimator, and (a) touches T0-1's frozen artifacts. Decide at T1-3.
- **D-032** 2026-07-16 (T1-3 prep) — **D-031 resolved by option (a): the fixtures now embed a linear-sRGB ICC profile.** Chosen by the product owner over the two cheaper alternatives. The file now states its own colour space instead of leaving every reader to guess; `starkit-io` reads the identity TRC and applies no curve, so decoded values are exactly `raw / 65535` — the ADU the truth catalog is written in, and the same units the oracle works in. Rejected: **(b)** an "assume linear" override on `starkit-io` — it would add a way to lie to the decoder about a file, and the file itself was the thing that was wrong; **(c)** never routing fixtures through `starkit-io` — it would leave the FR-1/FR-2 seam untested exactly where it matters. The profile is a standard matrix-shaper: sRGB primaries Bradford-adapted to the D50 PCS (identical to the reference sRGB profile) with an identity `curv` tone curve. "Linear sRGB" is an ordinary colour space, not an invention. Hand-built (~500 bytes of constants, no new dependency), and **the header's creation-date field is fixed at zero rather than read from the clock** — a timestamp there would make every regeneration a different file and break the manifest on every run (INV-2). **Blast radius, measured rather than assumed:** exactly 5 of 16 committed artifacts changed — the five `image.tiff`. Truth catalogs, params and the sky mask are byte-identical, and **all five oracle reports re-generated byte-identical** (recall 99.21 %, precision 99.87 %, centroid 0.056 px — unchanged to the last decimal), which is independent proof from a raw-pixel reader that only the container gained a tag and no pixel moved. Committed manifest updated; canonical seeds (D-007) untouched. Guarded end-to-end by `starkit-io/tests/fr1.rs::a_fixture_decodes_as_linear_not_as_srgb`, which asserts `decoded == raw/65535` exactly — a loose tolerance would hide the very bug this exists to catch, since an sRGB curve puts mid-grey at 0.214 rather than 0.5.
- **D-033** 2026-07-16 (T1-3) — **FR-2's numeric bars apply to `basic-5k` alone; the other four suites are pinned as regression floors. And the oracle cross-check is exceeded by 0.73 pp — in our favour.** Two rulings, both by the product owner, both with data. **(1) "All applicable suites"** cannot mean all five: measured, the oracle built in T0-2 *to calibrate this very bar* reaches 98 %/99 % on none of them (`dense-core` 88.1 %, `pairs` 65.8 %, `saturated` 96.6 %, `nightscape-fg` 91.6 %). docs/FIXTURES.md calls `basic-5k` the "primary metrics suite" and labels the rest deblending-stress, deblending-limit, saturation and nightscape. So `basic-5k` carries the bars — **recall 99.94 % (≥98), precision 99.92 % (≥99), median centroid 0.098 px (≤0.3), median FWHM error 5.77 % (≤10)**, all four met — and the others are asserted at floors below their measured values, which catches a regression without pretending 98 % is reachable on sub-resolution pairs. Explicitly **not** applied to `pairs`: the 0.3 px centroid bar. An unresolved blend's centroid genuinely lies between its components; at 0.414 px it is measuring the blend correctly. The oracle's better 0.086 px there is not a better instrument, it is a smaller sample — it matches 65.8 % where we match 69.3 %, so it scores only the easy ones. **(2) The oracle cross-check** allows Δrecall ≤ 0.5 pp and says larger deviations need a DECISIONS entry; this is it. We measure **99.94 % against the oracle's 99.21 %, Δ = 0.73 pp**, i.e. outside the window by being better. Cause is understood, not mysterious: a matched filter plus a tight centroid window beats DAOStarFinder's defaults, and our FWHM estimator (5.77 % vs its 9.90 %) is definition-based rather than a Gaussian fit to a Moffat. The test is therefore split rather than merely widened — falling 0.5 pp *below* the oracle still fails, since the oracle is the evidence these stars are findable at all, and drifting >2 pp above it also fails, since reporting what an independent instrument cannot see needs explaining too.
- **D-034** 2026-07-16 (T1-3) — **`nightscape-fg` precision is 35.7 % and stays that way until T1-5.** The zero-sum matched filter measures contrast against the neighbourhood, so ordinary sky beside a black tree silhouette reads as a source — exactly the effect D-019 identified in the oracle, which went from 59 % to 99.93 % precision only by excluding the foreground. The fix is a sky mask, and the owner's ruling is that it belongs to **T1-5**, not to `detect`. That is INV-1's own architecture: gating happens in exactly one place, the `gate` compositor, and adding a second mask check inside `detect` would scatter the invariant across two modules — precisely what INV-1 exists to prevent. Recall is unaffected (96.25 %): the sky stars are found, they are simply accompanied by silhouette-edge artifacts that nothing has yet been asked to remove. `detect_ac.rs` pins the recall and asserts precision stays **below** 0.90 — a tripwire, not a bar: if it ever rises without T1-5 landing, the silhouette response has changed and the reasoning here is stale. **T1-5 owes this suite a real precision assertion.**
- **D-035** 2026-07-16 (T1-4) — **Star-mask profile: smoothstep feather, union by maximum, `k=1.0` core with a `1.0 x FWHM` feather, saturated halo x2.** Four choices, each with a failure it avoids. **Smoothstep, not a linear ramp:** a linear feather has a discontinuous derivative at both ends of the band, and a 16-bit mask pushed through a Photoshop curve turns those corners into visible rings around every star. Smoothstep is C-1 at both ends; `mask::tests::the_feather_has_no_kink_at_either_end` bounds the second difference. **Union by maximum, not sum:** a mask is a coverage fraction and coverage does not add — two overlapping stars summing to 2.0 would clip into a flat-topped blob with a hard edge exactly where the sum crosses 1.0. Maximum is also commutative, so star order cannot change the output (INV-2), which is tested. **Core 1.0 x FWHM, zero by 2.0 x FWHM:** at 1 x FWHM from centre a Gaussian still holds ~6 % of its peak and a Moffat's wings reach further, so a 1 x FWHM disc alone would leave halos unmasked; 2 x FWHM covers the star without swallowing neighbours at the 8 px separations these fields actually have. **Saturated halo x2:** a clipped core has no measurable FWHM — the profile is flat where the information was — so `k x FWHM` describes the *hole*, not the star, and what needs covering is the bloom around it (FR-4's halo extension). Golden hashes are pinned over the **exported 16-bit codes**, not the f32 bits: the deliverable is the file, and an f32 hash would trip on a last-bit change no exported pixel could show. Verified to have teeth — a 1 % radius change fails it.
- **D-036** 2026-07-16 (T1-5) — **The gate composites by branch, not by blend; and D-034's debt is paid at a measured 8 px margin.** **(1)** The natural gated blend `out = input*(1-a) + modified*a` is *not* bit-identical at `a = 0`: `input + 0.0` turns `-0.0` into `+0.0`, and if a buggy operation produced a NaN then `NaN * 0.0` is NaN, so the poison crosses a gate that was supposed to be shut. INV-1 is a bit-level claim, so `Gate::composite` **branches**: `a == 0` copies. Both failures are tested directly (`negative_zero_survives_the_gate_unchanged`, `a_nan_in_the_modified_plane_cannot_cross_a_closed_gate`), and the invariant is proved on the real `nightscape-fg` frame with an absurd modified plane so a single touched pixel cannot hide in rounding. The vacuity guard matters as much as the invariant: a gate closed everywhere would pass trivially, so `the_gate_opens_on_sky_stars` asserts it opens on 0.5–30 % of the frame. Dilation uses a **disc, not a separable square**, which would extend the gate into the corners by root-2. **(2) D-034 paid:** `restrict_to_sky` lifts `nightscape-fg` precision from **35.7 % to 99.91 %**, against the oracle's 99.93 %. The margin is not a magic number — it is `detect::filter_reach_px()` (the kernel's own radius, 5 px at this seeing) **+ 3 px**, measured: surviving false positives sit a median 3.4 px from foreground with a p99 of 6.0, because a maximum triggered at the reach limit still gets a centroid a little further out. Sweep: 4 px -> 0.73, 6 px -> 0.99, 8 px -> 0.9991. Exposing the reach from `detect` is what stops the margin and the kernel drifting apart — a margin quietly smaller than the reach looks like it works and does not. **Cost, stated:** recall on this suite is **88.97 % vs the oracle's 91.63 %**, because our kernel truncates at 3-sigma where DAOFIND stops at 1.5-sigma. That same wider kernel is why `basic-5k` recall is 99.94 % against its 99.21 %. Buying precision with recall is the right way round here: a phantom star gets *operated on*, while a missed faint one is simply left alone.
- **D-037** 2026-07-16 (T1-6) — **Morphological reduction: disc erosion, and the shrink law is a measured trend, not the continuous ideal.** Method B is grayscale erosion (a local minimum over a structuring element) — what a Photoshop Minimum filter does, kept as the fast, familiar fallback while method A (resynthesis, T1-7) is the default. Three choices. **Disc element, not square:** a square erodes ~41 % further along the diagonals than the axes, quietly turning round stars diamond-shaped across a field; tested by `a_disc_element_erodes_no_further_diagonally_than_axially`. **Edges clamp, not zero-pad:** zero-padding would erode the frame border toward black — a dark rim exactly where FR-5 says this method must not make one; tested by `a_flat_field_is_unchanged_including_at_the_border`. **NaN cannot win the minimum:** a NaN is not smaller than anything, so it must not become the local min and erase its neighbourhood; tested. The half-max shrink law was where a first over-claim got caught: the doc asserted the continuous ideal (each isophote shifts inward by exactly the element radius, `R -> R-r`), but the digital disc under-reaches it and iterated passes compound sub-linearly — measured ~0.9 px/pass at radius 1, ~1.5-1.8 at radius 2, ~2.0-2.5 at radius 3. The doc now carries the measured table and the test asserts the robust claim (monotone in radius and in iterations) rather than a false linear formula; a precise 'FWHM reduced by X %' guarantee is method A's job (T1-7), not this one. **INV-1 discipline, first real exercise:** `morphological` erodes the *whole* plane and returns it — it has no idea where stars are and would happily erode a galaxy. Only `gate::composite` writes it into an image. Eroding globally then gating is deliberate and correct: a gated pixel's erosion legitimately *reads* neighbours from outside the gate (reading is not writing), and keeping the operation mask-ignorant is what makes INV-1 checkable in one place. Proved on the real `basic-5k` and `nightscape-fg` at zero tolerance outside the gate — a reduction touches no nebulosity and no trees.
- **D-038** 2026-07-16 (T1-7) — **Resynthesis reduction rebuilds each star by radial resampling of its own pixels, not a fitted model.** Method A, the default (FR-5). Per star: annulus-median background -> inpaint the core disc with that background (removing the original star) -> re-add the star's own profile sampled at `d/r` from centre, so `new(d) = old(d/r)` and the half-maximum lands at `r*R`: **new_fwhm = r * old_fwhm** in the continuous limit, peak preserved (a reduced star stays as bright, becomes tighter). Resampling real pixels rather than a Gaussian-or-Moffat fit is the whole point: it carries the true PSF shape, ellipticity and — per channel — colour, with none of the model mismatch T0-2 measured at +14 % on Moffat stars. **Bilinear interpolation is deliberate, not lazy:** it is a convex combination so it cannot overshoot, which makes FR-5's 'no ringing' free — a higher-order kernel would tighten small stars but could ring. The cost is a sampling floor: bilinear broadens the smallest cores, so at `r = 0.5` the achieved FWHM ratio is 0.537 (+7.4 %, inside the +/-10 % bar) and at `r = 0.4` it reaches +10.5 % on this 3.2 px seeing — a sub-Nyquist limit, documented like `detect`'s, not tuned away. **Only positive signal is re-added** (a resampled noise dip would build a faint dark speck). **Saturated stars are protected by default:** a clipped core has no measurable profile, so resampling a flat-topped hole would make a smaller hole, not a smaller star. All five FR-5 bars met on the real `basic-5k`, graded through the full detect -> mask -> gate -> resynthesis pipeline: FWHM within +/-10 %, star count preserved (98.9 % of centroids keep >=70 % of peak), no sub-background dark halo, outside-gate bit-identity on `basic-5k` *and* `nightscape-fg`, and determinism with a pinned golden hash (teeth verified). `r = 0` is the starless path (FR-7's building block): inpaint stands alone, nothing re-added. **Deferred honestly:** the inpaint is a flat annulus-median fill, so the removed-star region is noiseless where the surroundings are not. Invisible under a shrunk star and at background level (no dark halo), but on a real wide-field image a noise-matched fill may be wanted; that is a real-corpus refinement (D-027), not a fixtures one.
- **D-039** 2026-07-16 (T1-8) — **The CLI: versioned preset, stable exit-code table, and no clock in any hashed output.** `starkit process | inspect | batch`, orchestration only — it wires `starkit-io` to `starkit-core` and owns nothing photometric. Four choices worth recording. **(1) Preset schema tagged and all-defaults-valid:** a preset is the entire parameter set in one JSON with a `starkit-preset/1` tag; every field defaults, so `{}` is a valid preset and a caller writes only what they change, while a foreign schema is rejected rather than mis-read. The preset restates the core structs' defaults rather than importing them, so a saved preset is self-contained and explicit — guarded by `preset_defaults_match_the_core_param_defaults`, which fails if a core default drifts. **(2) Exit codes are a contract** (`src/exit.rs`, documented in `starkit-cli/CLAUDE.md`): 0 OK, 2 USAGE, 3 IO, 4 DECODE, 5 UNSUPPORTED, 6 PROCESSING, **7 BATCH_PARTIAL** — the one FR-8 cares about, since a batch that hits a poisoned file must finish the rest and exit non-zero so a script can tell 'all done' from 'some skipped' without parsing logs. Asserted through the *compiled binary* in `tests/cli_ac.rs`, not just the library. **(3) No wall clock reaches a hashed output** (INV-2 / FR-8's hash-identical re-run): the per-image report deliberately carries **no timing** — a report is an output too, and a timestamp would defeat a byte-comparison — and batch inputs are processed in **sorted** order so filesystem enumeration cannot reorder them. Elapsed time goes to the log, where it informs without being pinned. Verified: two runs of a 100-image batch produce byte-identical reduced TIFFs. **(4) Reduction runs per channel** with the geometry fixed once on the mean-of-RGB plane (detection, masking, gate all shared), so a star's colour survives and the three channels cannot drift out of registration; INV-1 holds for the whole CLI because reduction still only reaches an image through `gate::composite`. All three FR-8 bars met on the compiled binary: a 100-image batch runs unattended, a re-run is hash-identical, and one poisoned file is reported+skipped without aborting (exit 7). **Note:** FR-8's own AC is fully met; the broader G1 sign-off still waits on the real corpus (D-027).
