# StarKit — 星空摄影自动星点处理软件

**功能规格说明书（PRD）· v1.0 · 2026-07-16 批准**
**状态：已批准（2026-07-16）。实现以英文版 PRD.md 为准；变更须在 docs/DECISIONS.md 登记。**
**读者：产品负责人（MingJiang）、核心用户（职业星空摄影师）、Claude Code（实现方）。**

术语对照：star reduction = 缩星 · star enhancement = 提亮星点 · nightscape = 星野（天空+地景）· deep-sky = 深空 · diffusion = 柔焦

## 1. 问题陈述

职业星空摄影后期中，两类重复性操作耗时最多：

1. **缩星。** 叠加与拉伸之后，星点相对星云/银河显得臃肿。手工流程（Photoshop：色彩范围选区 → 手工修蒙版 → 最小值滤镜 → 修补黑边和断裂星点；PixInsight：建星点蒙版 → MorphologicalTransformation → 修补瑕疵）单张耗时 15–40 分钟，且系列作品之间难以保持一致。
2. **提亮星点。** 让主要亮星突出、星座结构清晰可读：复制图层 → 高斯模糊 → 滤色混合 → 手绘蒙版只保留主星 → 恢复过曝星核的颜色。同样繁琐，同样不可复现。

两个痛点的根因相同：**真正的瓶颈是生成一张精确、无瑕疵的星点蒙版。** 有了高质量蒙版之后，缩星与提亮都只是廉价的参数化运算。另有一个星野摄影特有的问题：任何自动星点操作都不能碰地景（树、山脊线、建筑）——粗糙的工具会把地景高光当成星点。

**目标：** 一款桌面工具，自动完成 星点检测 → 天空/地景分离 → 星点蒙版 → 参数化缩星/提亮，支持确定性的批处理，并能无损往返现有 Photoshop 工作流。目标是把 20–40 分钟的手工任务压缩到 1 分钟以内 + 人工复核。

## 2. 目标用户与工作流位置

- **核心用户：** 职业星野/深空摄影师。以 Photoshop 为中心。处理对象为叠加、拉伸后的 **16-bit TIFF**，通常 2400 万–1 亿像素。
- **次级用户：** MingJiang——CLI 批处理、流水线集成、维护。

工作流位置：在叠加/拉伸（Sequator / Siril / PS）**之后**，最终调色**之前或并行**。因此工具导出的图层与蒙版必须能干净地回到 PS（16-bit、保留 ICC、像素对齐）。

## 3. 目标与非目标

**目标（v1.x）**

- G1——自动星点检测 + 天空/地景分离。
- G2——参数化缩星，硬保证：星点蒙版之外的像素一律不动。
- G3——选择性提亮：辉光、亮度提升、星点颜色恢复、可选柔焦（Softon 风格）模拟。
- G4——确定性批处理：同输入 + 同参数 → 输出逐位一致。
- G5——可导出的中间产物供 PS 合成：星点蒙版、无星层（starless）、纯星点层。

**非目标（v1）**

RAW 解码/去马赛克 · 叠加/对齐 · 渐变与光害去除 · 降噪 · 通用修图 · 赤道仪相关 · 移动端。（部分可作为 v2+ 候选；现在明确排除，以保证阶段关口的严肃性。）

## 4. 处理流水线（总览）

```
输入 TIFF ──▶ 线性化（ICC → 线性工作空间）
        ──▶ 天空/地景分割（自动 + 导入/手绘蒙版）
        ──▶ 背景建模（分块 sigma-clip 估计）
        ──▶ 星点检测 → 星表（x, y, 流量, FWHM, 峰值, 饱和标记, 分级）
        ──▶ 星点蒙版（分级、羽化）
        ──▶ [缩星] ──▶ [提亮] ──▶ 重合成
        ──▶ 反线性化 ──▶ 输出 16-bit TIFF（+ 可选：蒙版 / 无星层 / 纯星层）
```

所有光度运算在线性光下进行；gamma/ICC 只在边界处理。所有星点操作受 `天空蒙版 ∧ 星点蒙版` 门控——这是结构性不变量，不是约定俗成。

## 5. 功能需求

### FR-1 图像输入输出

- **输入：** TIFF 8/16-bit RGB（单图层）、PNG、JPEG。识别内嵌 ICC（sRGB / AdobeRGB / ProPhoto）。测试上限 1 亿像素（约 100 MP，D-042）。
- **输出：** 默认 16-bit TIFF。保留 ICC 与 EXIF。默认绝不覆盖输入文件；临时文件 + 原子重命名。
- **验收：** 空跑（全部操作关闭）输出与输入 TIFF16 逐像素一致。损坏/截断的输入以干净报错失败，绝不产生半成品文件。

### FR-2 星点检测与 PSF 测量

- 分块背景 + 噪声估计（sigma-clip）；阈值 + 局部极大值检测；亚像素质心；逐星 FWHM（椭圆拟合——保留椭率供后续星形修复用）；饱和标记；近邻双星去混叠；按流量与 FWHM 分级（大/中/小）。
- 输出：星表（JSON + 内存结构），是所有下游操作的基座。
- **验收（合成金标准星场，生成器随仓库提交）：** SNR ≥ 5 的星召回率 ≥ 98%；精确率 ≥ 99%；质心误差中位数 ≤ 0.3 px；FWHM 误差中位数 ≤ 10%。
- **验收（真实语料，摄影师提供 ≥ 20 张含难例的 TIFF）：** 结构化目视 QA 清单通过；与 Python oracle（photutils DAOStarFinder）交叉验证在约定容差内。

### FR-3 天空/地景分割

- **自动模式：** 利用星野结构特征的启发式分割（天空 = 平滑渐变 + 点源；地景 = 暗、高纹理、自底部连通）。输出天空蒙版 + 置信度图。
- **手动兜底（Phase 1 必须交付）：** 导入外部灰度蒙版（PNG/TIFF），可在 PS 里画。GUI 画笔在 Phase 2。自动模式尽力而为，手动路径才是保证。
- **硬性不变量：** 天空蒙版之外零星点操作——结构性强制并有测试覆盖。
- **验收：** 干净地平线测试图上与人工标注 IoU ≥ 0.95；复杂地平线（树）永远可用导入蒙版纠正；不变量测试证明地景像素与输入逐位一致。

### FR-4 星点蒙版生成（本身即是交付物）

- 从星表生成软蒙版：单星半径 = k × FWHM，羽化可调；饱和星做光晕外扩；分级蒙版（大/中/小）+ 合并版。
- 导出 16-bit 灰度 TIFF/PNG，与原图像素对齐——可直接作为 PS 图层蒙版。*仅此一项就替代了最繁琐的手工步骤，是第一个用户可见的胜利。*
- **验收：** 摄影师确认导出的蒙版在她自己 ≥ 5 张作品的 PS 流程中可直接使用。

### FR-5 缩星

- **方法 A——重建法（默认）：** 逐星处理：估计局部背景（环带 + 修复填充）→ 提取星点轮廓 → 按缩减系数 r ∈ [0, 1] 以 PSF 模型缩放后回贴。星点颜色取自未过曝的核心环带。避免最小值滤镜的经典瑕疵（黑边、星点断裂、星云被侵蚀）。
- **方法 B——形态学（备选）：** 蒙版内腐蚀，N 次迭代。速度快，效果贴近摄影师熟悉的样子。
- 参数：强度 r；尺寸-亮度权衡；分级选择（如只缩中小星、保留主星）；保护饱和星开关；星云保护选项（结构感知排除，用真实语料调参）。
- **验收：** 膨胀后星点蒙版之外的像素与输入逐位一致；星点数量不变（r = 0 之外不删星）；实际 FWHM 缩减中位数与设定值误差 ≤ ±10%；金标准 diff 无振铃/黑边；多次运行确定性一致。

### FR-6 提亮星点

- **选择方式：** 按流量取前 N、按流量百分位、或显式按分级。（P3 选项：天体测量定位 + 星表匹配，定向提亮星座主星——见 §9。）
- **效果（可独立开关）：** 柔和辉光（高斯 bloom，线性光下滤色合成）；星核亮度软膝提升（"保护高光"开启时不产生新过曝）；过曝星核颜色恢复（色相取自未过曝环带）；柔焦滤镜模拟预设（Softon 风格，半径随星点流量缩放）；合成衍射星芒（默认关闭）。
- **验收：** 高光保护开启时新增过曝像素为零；效果限制在所选星点的膨胀蒙版内；确定性一致。

### FR-7 无星/纯星分解

- 缩星路径取 r = 0 即得无星层；纯星层 = 输入 − 无星层（线性空间）。两者均可导出 16-bit TIFF 供 PS 合成或第三方使用。
- **验收：** 无星层 + 纯星层重合成的 RMS 误差 ≤ 16-bit 下 1 LSB。

### FR-8 批处理与预设

- CLI：按版本化预设（JSON schema）处理整个文件夹；稳定退出码；逐图 JSON 报告（星点数、参数、耗时、警告）。
- **验收：** 100 张批处理无人值守完成；重跑输出哈希一致；单张失败不中断整批（记录、跳过）。

### FR-9 GUI（Phase 2）

前后对比滑块 · 100% 缩放平移 · 彩色蒙版叠加显示 · 参数滑块 + 实时代理预览（4K 代理上单次更新 ≤ 200 ms）· 补选/排除画笔（回馈 FR-3）· 批处理队列 · 预设管理 · 中英双语界面。

## 6. 非功能需求

- **性能：** 6100 万像素 16-bit TIFF 全流程（检测 + 缩星 + 提亮）目标 ≤ 10 s / 硬上限 20 s（8 核桌面 CPU；实测 8.3 s，D-040）。基线由 Phase 1 基准测试确立并锁定为回归预算。1 亿像素（约 100 MP）为已测尺寸上限（D-042），约 12 s——超过 10 s 目标但在 20 s 硬上限内。
- **内存：** 峰值约为解码后（f32）图像的 4 倍 + O(星表)——61MP 与 100MP 实测均为 4.1×（D-042）。原定 ≤ 2× 目标推迟至内存优化（缓冲复用/分块流水线）之后，与 rayon 并行同属未来工作（D-040）。
- **确定性：** 输出不依赖时钟或线程调度顺序；有随机性处固定种子。
- **可靠性：** 输入文件永不被修改；运行中崩溃不留半成品（临时文件 + 原子重命名）。
- **可移植性：** Windows 10+ 与 macOS 13+ 为一等目标（待 Q1 确认）；Linux 在 CI 中保持常绿。
- **许可：** 仅使用宽松许可依赖（MIT/Apache/BSD）。**v1 不引入 StarNet 权重**——其许可为非商业且来源受限；基于 ML 的去星推迟到 Phase 3，用自训练或合规授权的模型。

## 7. 架构与技术栈（提案）

- **Rust workspace：** `starkit-core`（纯算法，尽量无 I/O）· `starkit-io`（TIFF/ICC/EXIF）· `starkit-cli` · `starkit-gui`（Phase 2；egui vs. Tauri 在 Phase 1 出关时按预览延迟、打包体积、CJK 字体渲染三项决策）。
- **并行：** rayon 分块流水线；SIMD 热点仅在性能分析后引入；GPU（wgpu）是 Phase 3 的问题。
- **Python oracle（`oracle/`）：** photutils（DAOStarFinder）+ scipy 参考实现，在金标准夹具上交叉验证检测、质心、FWHM——与 NRS 同一套路。
- **金标准夹具：** 随仓库提交的合成星场生成器（Moffat/高斯 PSF、真实噪声模型、饱和、近邻双星、合成地景剪影），附期望星表与期望输出哈希。

## 8. 测试策略与完成定义

逐模块单元 + 属性测试 · 金标准图像 diff（蒙版之外零容差）· oracle 交叉验证 · 确定性测试（双跑哈希相等）· 性能基准与基线记录 · **摄影师在环 QA：** 每阶段一份结构化打分表，另加 5 张对照基准——她把工具输出与自己手工 PS 结果对比（通过标准 = 在时间-质量权衡上 ≥ 4/5 张评为不劣于手工）。

每阶段的完成定义 = 全部验收标准通过 + 基准在预算内 + QA 表签字。No green test, no done。

## 10. 开放问题——Phase 0 出关的阻塞项

- **Q1（关键）：** 摄影师的操作系统与形态偏好——独立 GUI、CLI 批处理，还是 Photoshop 插件才是真正的优先级？决定 Phase 2 的形态。
- **Q2（关键）：** 她能否提供 ≥ 20 张有代表性的 16-bit TIFF 作为测试语料，含难例（地平线有树、银河核心密集星场、强星云、重度饱和亮星）？她的星野 : 深空比例大致是多少？
- **Q3：** 完整的无星图是她想要的艺术交付物，还是只是内部中间步骤？（影响 FR-7 的打磨优先级。）
- **Q4：** 提亮口味——她用不用星芒/重辉光，还是只要克制的效果？给一两张她心目中"理想成片"的参考图，能锚定 FR-6 的默认参数。
- **Q5：** 分发方式——只给她私用，还是将来可能公开发布？（影响许可严格度、代码签名/公证、UI 打磨预算。）
- **Q6：** 她的机器有无独立 GPU？（只影响 Phase 3 规划。）

## 11. 风险与缓解

- **复杂地平线的自动天空分割**（树枝映衬天空）确实很难 → 手动蒙版导入在 Phase 1 交付作为保证路径；自动模式只是叠加的便利层，永不成为依赖。
- **深空片中星云被误判为星点** → 分级阈值 + 星云保护选项，G1 出关前用真实语料验证。
- **"好"有主观成分** → 每阶段摄影师在环 QA 表 + 5 张手工 vs 工具对照基准，保证目标不漂移。
- **向通用天文修图软件的范围蔓延** → §3 非目标具有契约效力；新增内容一律单独设关。
